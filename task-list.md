# CodexSwitch 任务列表

文档状态：草案 v0.1

本任务列表基于当前 `requirement.md` 与 `design.md` 生成，目的是给首版实现提供一个清晰的执行顺序。等你下一轮修改需求文档后，这份任务列表应该重新生成一次。

- [ ] 基于最终版需求回刷 `design.md`
- [ ] 基于最终版需求回刷 `task-list.md`

## 1. 项目初始化

- [ ] 使用 Wails 创建 `Vue 3 + TypeScript` 项目骨架
- [ ] 接入 Vuetify 3
- [ ] 接入 Pinia
- [ ] 约定前后端目录结构与命名规范
- [ ] 建立统一的错误提示与消息通知机制

交付标准：

- 应用可以本地启动
- 前端与 Go 后端可以正常互调

## 2. 设置与路径管理

- [ ] 实现默认 `codexHomePath` 推导
- [ ] 实现 `settings.json` 读写
- [ ] 实现设置弹窗
- [ ] 支持修改目标 Codex 配置目录并持久化
- [ ] 路径修改后自动触发重扫

交付标准：

- 重启应用后仍能记住上次设置的路径
- 更改路径后列表和激活状态会刷新

## 3. 后端基础能力

- [ ] 实现应用配置目录初始化
- [ ] 实现托管配置目录扫描
- [ ] 实现文件安全写入工具
- [ ] 实现 `contentHash` 计算
- [ ] 实现统一日志与错误包装

交付标准：

- 可以稳定读写托管仓库
- 文件操作失败时能返回明确错误

## 4. `auth.json` 与 `config.toml` 解析

- [ ] 实现 `auth.json` 最小结构体
- [ ] 实现账号类型识别规则
- [ ] 实现官方 `auth.json` 标准化
- [ ] 实现 CLI 扁平 JSON 转标准官方 `auth.json`
- [ ] 实现 CLI 敏感字段缺失校验
- [ ] 实现官方 JWT 元数据解析
- [ ] 实现 API Key 元数据提取
- [ ] 实现 `config.toml` 关键字段解析
- [ ] 实现 `profile_id` 生成规则

交付标准：

- 能正确识别 `conf/codex` 和 `conf/api`
- 能生成稳定的配置身份标识

## 5. 当前配置扫描与自动同步

- [ ] 实现当前目标目录扫描
- [ ] 让当前目录扫描兼容 CLI 扁平 `auth.json`
- [ ] 实现启动自动导入逻辑
- [ ] 实现启动自动更新逻辑
- [ ] 实现激活配置识别
- [ ] 实现异常配置状态标记

交付标准：

- 启动后可自动识别当前配置
- 当前配置未托管时会自动导入
- 当前配置已托管但有变化时会自动更新

## 6. 托管配置仓库

- [ ] 定义 `meta.json` 结构
- [ ] 实现新增托管配置
- [ ] 实现更新托管配置
- [ ] 实现读取单个托管配置
- [ ] 实现读取全部托管配置
- [ ] 实现删除托管配置

交付标准：

- 托管配置目录结构固定且可重建
- 配置列表可以稳定加载

## 7. 导入当前配置

- [ ] 实现“导入当前配置”后端接口
- [ ] 支持同身份配置更新而不是重复创建
- [ ] 导入后刷新列表

交付标准：

- 用户可以手工导入当前配置
- 重复导入不会产生重复记录

## 8. 导入官方账号文件

- [ ] 实现官方账号文件选择与导入后端接口
- [ ] 支持标准官方 `auth.json` 导入
- [ ] 支持 CLI 扁平 JSON 导入并标准化保存
- [ ] 固定使用官方 `config.toml` 模板生成托管配置
- [ ] 支持同身份配置更新而不是重复创建
- [ ] 导入成功后刷新列表但不自动切换
- [ ] 接入前端工具栏按钮与提示

交付标准：

- 用户可以通过文件选择框导入官方账号文件
- CLI 样例导入后仓库中保存的是标准官方 `auth.json`
- 重复导入不会产生重复记录

## 9. API 配置新增与编辑

- [ ] 将 `conf/api` 样例整理为内置模板
- [ ] 实现新增 API 配置表单
- [ ] 实现 API 配置模板渲染
- [ ] 实现 API 配置保存
- [ ] 实现 API 配置编辑
- [ ] 实现 API Key 掩码展示

交付标准：

- 用户可以新增 API 配置
- 用户可以编辑已有 API 配置
- 列表中不会泄露完整 API Key

## 10. 配置切换

- [ ] 实现切换前当前配置保护逻辑
- [ ] 实现切换前自动导入/自动更新
- [ ] 实现目标配置写入目标目录
- [ ] 实现切换后的重扫与状态刷新
- [ ] 实现重复切换同一配置的 no-op 优化

交付标准：

- 能在官方配置和 API 配置之间切换
- 切换后激活状态正确刷新

## 11. 配置删除

- [ ] 实现额度拉取状态缓存

建议按下面顺序推进：

1. 项目初始化
2. 设置与路径管理
3. `auth.json` / `config.toml` 解析
4. 当前配置扫描与自动同步
5. 托管配置仓库
6. 导入当前配置
7. 导入官方账号文件
8. API 配置新增与编辑
9. 配置切换
10. 配置删除
11. 前端主界面
12. 官方额度信息
13. 测试与打包
